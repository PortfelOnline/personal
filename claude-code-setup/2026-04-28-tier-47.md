# Tier 47: Enhanced DeepSeek routing — Weighted scoring + Retry with fallback

**Дата**: 2026-04-28
**Статус**: ✅ Реализовано
**Цель**: Улучшить роутинг моделей DeepSeek (flash vs pro) — заменить keyword-based эвристику на weighted scoring, добавить exponential backoff retry и fallback между моделями.

## Зачем

Старый роутер (`route_model()`) проверял keywords по принципу "если слово есть → pro". Это давало false positives на коротких запросах, содержащих общие слова. Не было retry-логики — любой HTTP 429/5xx ронял весь чат. Не было fallback — если pro падал, не было переключения на flash.

## Архитектура

### Weighted scoring (`deepseek_api.py:route_model()`)

```
route_model(message, system, model_hint, temperature, json_mode)
  │
  ├── PRO_KW = ["архитектур", "рефакторинг", "аудит", "миграци", ...] → +2 each
  ├── FLASH_KW = ["hello", "hi", "погода", "время", ...] → -1 each
  │
  ├── Длина > 1000 chars → +1.5
  ├── Длина > 500 chars → +0.5
  ├── JSON mode → +0.5
  ├── system prompt > 200 chars → +0.5
  │
  └── score >= 2.0 → deepseek-v4-pro
      score < 2.0 → deepseek-v4-flash
```

Порог 2.0 означает: нужно минимум 1 PRO_KW (2.0) или комбинация длины + JSON (2.0+), чтобы получить pro. Это уменьшает количество ложных назначений pro.

### Retry logic (`deepseek_api.py:_call_with_retry()`)

```
_call_with_retry(api_func, *args, max_attempts=3, fallback_model=None)
  │
  ├── Попытка 1: model=pro
  │   ├── HTTP 429/5xx → sleep(2^1 * 1.0s = 2s) → attempt 2
  │   ├── ConnectionError → sleep(2^1 * 0.5s = 1s) → attempt 2
  │   └── Success → return
  │
  ├── Попытка 2: model=pro
  │   ├── HTTP 429/5xx → sleep(2^2 * 1.0s = 4s) → attempt 3
  │   └── ...
  │
  ├── Попытка 3: model=pro
  │   └── fail → если fallback_model:
  │       └── retry 3 times with fallback_model
  │       └── всё fail → raise
  │
  └── Всего: до 6 API calls (3 pro + 3 fallback flash) за один _api_call_with_tools
```

### Agent.py dispatch

`_api_call_with_tools()` и `_api_call_with_tools_stream()` — обе функции получили inline retry loop:
- 3 попытки на текущей модели
- При исчерпании: fallback на flash (если текущая модель pro)
- Логирование каждой попытки через `log(f"[Retry] ...")`

### `chat()` в `deepseek_api.py`

Теперь вызывает `_call_with_retry()` вместо прямого `_api_call()`. Non-streaming — retry наиболее полезен для не-streaming вызовов.

### `stream_chat()` — НЕ тронут

Streaming retry сложнее (нельзя перезапросить середину стрима). Пока без изменений.
